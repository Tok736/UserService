# UserService

Микросервис профилей и доменных связей репетиторской платформы. Отвечает за людей и
отношения между ними; аутентификация живёт в отдельном AuthService.

## Модель данных

Ключевое решение: **отдельной таблицы студентов нет**. Репетитор, ученик и родитель —
это строки в одной таблице `user`. «Ученик» и «репетитор» — это *роли в связи*, а не тип
записи, поэтому один человек одновременно бывает учеником у одного и репетитором у другого.
Связи между людьми — самоссылающийся M:N (`user_relation`), а учебный контекст (предметы,
уровень, заметки, теги) висит **на связи**, потому что у разных репетиторов он разный.

Гибридный ученик: сначала это управляемая карточка без аккаунта (`account_status = managed`),
позже она активируется в полноценный аккаунт через приглашение — карточка та же, история цела.

| Таблица | Назначение |
|---|---|
| `user` | человек: профиль, контакты, статус аккаунта |
| `user_relation` | связь человек↔человек с типом и учебными метаданными |
| `student_group` / `group_membership` | группы учеников и их состав |
| `invitation` | приглашение карточки к активации |
| `consent` | версионированные согласия на обработку ПД, в том числе родительские |
| `outbox_message` | транзакционный outbox |
| `processed_message` | дедупликация входящих сообщений |
| `audit_log` | журнал доступа и изменений |

## Как это работает

**Транспорт.** CRUD для фронтенда — синхронный HTTP (требование проекта), межсервисное
взаимодействие — RabbitMQ. И то и другое живёт в одном процессе: подписчики шины и
публикатор outbox должны ходить в ту же базу, что и API.

**Права.** `user_id` и scopes берутся из JWT: в Auth за ними никто не ходит и роли не кешируются.

**Мультиарендность** — это не система прав, а обычная проверка связей в сервисном слое:
`is_visible` пускает к профилю, если это ты сам, человек с прямой связью или ученик,
которого вы ведёте вместе с другим репетитором. Всё остальное фильтруется по связям.

**Транзакционный outbox.** Доменная запись, событие в outbox, отметка о дедупликации и
запись в audit_log коммитятся **одной транзакцией** — поэтому события не теряются и не
уходят в шину при откате. Фоновый `OutboxPublisher` публикует их порядком «сначала
публикация, потом отметка»: доставка at-least-once, а получатель дедуплицирует по `message_id`.
Единственное исключение — команда `auth.identity.create`: она несёт пароль, которому нельзя
лежать в базе даже временно, поэтому идёт в шину напрямую.

**Идемпотентность.** Все обработчики событий Auth дедуплицируются по `message_id` в той же
транзакции, что и доменные изменения.

**Сага активации:**

1. Репетитор приглашает карточку → `invitation` + событие `user.invitation.sent` (сырой токен
   уходит только в событии, в базе лежит его HMAC-дайджест).
2. Приглашённый принимает → собираем согласия (для несовершеннолетнего обязательно
   родительское) → команда `auth.identity.create` в Auth.
3. Auth создаёт аккаунт и публикует `auth.user.registered`.
4. UserService проставляет `user_id`, переводит карточку в `active`, закрывает приглашение
   и публикует `user.activated`.

**Единый формат участника обучения.** События `user.relation.*` и `student.group.*` несут
одинаковый блок `participant`, чтобы ScheduleService обрабатывал индивидуальные и групповые
занятия одним кодом.

## Персональные данные

На платформе есть несовершеннолетние, поэтому:

- контакты и заметки шифруются на уровне полей (Fernet), поиск по точному совпадению идёт
  через слепой индекс (HMAC), а не расшифровкой;
- право на удаление — это **обезличивание**, а не физическое удаление: строка остаётся,
  иначе рассыплется история уроков и платежей в других сервисах;
- согласия версионированы, родительское согласие обязательно для активации карточки ребёнка;
- доступ к чужим профилям пишется в `audit_log`, значения ПД в журнал не попадают — только
  имена изменённых полей.

## Запуск

```bash
pip install -r requirements.txt

cp config.json /edcurve/webconf/user_service/config.json   # локальный конфиг сервиса
python src/scripts/generate_keys.py                        # ключи шифрования полей
python src/scripts/create_database.py                      # база и таблицы
python src/main.py
```

Секреты и адреса инфраструктуры приезжают из общего `/edcurve/webconf/config.json`.
Health-check'и: `/health/live` и `/health/ready`, метрики Prometheus: `/metrics`.
